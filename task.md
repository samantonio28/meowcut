Укорачиватель ссылок

Необходимо реализовать сервис, который должен предоставлять API по созданию сокращенных ссылок следующего формата:
- Ссылка должна быть уникальной и на один оригинальный URL должна ссылаться только одна сокращенная ссылка.
- Ссылка должна быть длинной 10 символов
- Ссылка должна состоять из символов латинского алфавита в нижнем и верхнем регистре, цифр и символа _ (подчеркивание)

Сервис должен быть написан на Go и принимать следующие запросы по http:
1. Метод Post, который будет сохранять оригинальный URL в базе и возвращать сокращённый
2. Метод Get, который будет принимать сокращённый URL и возвращать оригинальный URL

Решение должно быть предоставлено в «конечном виде», а именно:
- Сервис должен быть распространён в виде Docker-образа 
- В качестве хранилища ожидается использовать две реализации. Какое хранилище использовать, указывается параметром при запуске сервиса.  
    - Первое это postgresql.
    - Второе - самостоятельно написать пакет для хранения ссылок в памяти приложения.
- Покрыть реализованный функционал Unit-тестами

Результат предоставить в виде публичного репозитория на github.com

В процессе собеседования-ревью посмотрим:
- Как генерируются ссылки и почему предложенный алгоритм будет работать; насколько он соответствует заданию и прост в понимании.
- Как раскиданы типы по файлам, файлики по пакетам, пакеты по приложению: структуру проекта.
- Как обрабатываются ошибки в разных сценариях использования
- Насколько удобен и логичен сервис в использовании
- Как сервис будет себя вести, если им будут пользоваться одновременно сотни людей (как например youtu.be / ya.cc)
- Что будет, если сервис оставить работать на очень долгое время
- Общую чистоту кода

структура проекта должна быть следующей
в корне должен лежать файл doc.yaml, в котором будет лежать спецификация нашего сервиса (два метода, указанные в задании). Также в корне проекта должна быть папка configs в которой будут находиться конфиг для базы данных postgres (там будут переменные host port password db и все такое)
Также в корне проекта будет лежать папка pkg/utils/, в которую можно положить какие-то полезные функции (например генератор ссылки из оригинальной ссылки), какая-нибудь функция makeResponse, в которой выставляется content-type = application/json и json.Decoder(r).Response(w) происходит примерно (то есть делается отправка результата))
Также в корне проекта будет лежать папка cmd, в которой будет лежать файл main.go, который и будет запускать весь проект (в нем будет всего лишь что-то в духе func main () { delivery.App(domain.NATIVE) }) (или domain.PG от слова PostgreSQL) (это будут константы две, которые обозначают, как хранить ссылки, в памяти или в постгрессе)
Также в корне проекта будет лежать папка internal. Мы пишем проект по чистой архитектуре, в которой 4 слоя: delivery (обработчики), usecase (юзкейсы), service (репозиторий умеющий хранить ссылки и возвращать их), domain (доменные сущности, константы, ошибки, интерфейсы). В папке internal будут следующие папки:
- domain - здесь будут лежать доменные сущности по типу Link, LinkRepository (интерфейс репозитория), LinkCutter (интерфейс укорачивателя ссылок), ошибки, константы NATIVE = "native" и PG = "pg". Все сущности стараемся по своим файлам раскидать, нет ничего такого чтобы на интерфейс LinkCutter с одним методом был отдельный файл.
- service - здесь будет лежать 2 реализации интерфейса LinkRepository (первый будет nativeLinkRepo (хранение в map[string]string), второй pgLinkRepo (хранение в постгрессе)), а также LinkMeowCutter (реализация интерфейса LinkCutter). По поводу pgLinkRepo: понадобится файл миграций бд - его можно спокойно создать в папке migrations в корне проекта, дальше взаимодействовать с ним будет docker-compose. Скорее всего понадобится одна таблица, где ключом будет укороченная ссылка, а значением будет оригинальная ссылка
- usecase - здесь будет два файла, каждый по одной струкруре: конструктор юзкейса как структурки, в которую передали внутрь реализацию репозитория (и еще возможно LinkCutter) и указатель на логгер (логгер будет инициализироваться на уровне app), затем метод, соответствующий юзкейсу, где используются репозиторий и возможно укорачиватель.
- delivery - здесь будет код, нужный для общения по http.
